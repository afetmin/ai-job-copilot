# Workspace Layout Optimization Design

## 目标
- 把 `frontend/src/components/workspace/interview-pack-form.tsx` 从“信息堆叠页”重构成更符合操作顺序的工作台。
- 按“输入材料 -> 调整执行参数 -> 提交并观察状态”的流程重排模块，降低用户视线来回跳转成本。
- 保留当前纸面装裱式视觉语言、现有字段名、提交流程、校验逻辑和跳转行为。

## 当前问题
- 左侧和右侧同时承载输入、执行、说明三类信息，模块职责混杂。
- 提交动作、生成状态、系统说明都在右栏堆叠，执行区缺少明确的优先级。
- 顶部只有静态摘要，没有把当前页明确表达成 4 个连续步骤。
- 桌面端阅读路径不稳定，用户会在“左侧填内容”和“右侧看说明”之间频繁切换。

## 设计结论
- 采用“左侧主流程 + 右侧固定执行栏 + 顶部轻量步骤导航”。
- 左侧只保留输入和提交前确认模块，形成连续编辑区。
- 右侧固定栏只保留执行控制、状态反馈和精简后的系统边界。
- 移动端取消固定侧栏，按步骤顺序回退为单列布局。

## 信息架构

### 顶部区域
- 保留 `AppShell` 页面抬头与现有视觉风格。
- 在抬头下方新增一条轻量流程导航，展示：
  - `01 输入简历`
  - `02 输入岗位`
  - `03 调整参数`
  - `04 提交生成`
- 流程导航只做阶段提示和状态摘要，不引入新交互，不改变表单提交流程。

### 左侧主流程区
- `01 简历材料`
  - 保留简历文本输入和 PDF 上传。
  - 文案强调这里只放候选人材料，不混入岗位信息。
- `02 岗位上下文`
  - 保留岗位文本输入和 PDF 上传。
  - 文案强调这里只放 JD、职责要求和评估维度。
- `03 生成前确认`
  - 收束当前分散在多个卡片里的辅助说明。
  - 用轻量摘要显示本次是否已提供简历文本/PDF、岗位文本/PDF，以及提交后会发生的动作。

### 右侧固定执行栏
- `生成参数`
  - 保留 `targetRole` 和 `questionCount`。
  - 继续作为本次请求的控制区，不承载核心输入内容。
- `提交任务`
  - 保留提交与重置按钮。
  - 维持原有 `FormData` 提交结构和成功后的路由跳转。
- `生成状态`
  - 继续由 `GenerationStatusCard` 展示当前阶段、说明和下一步。
  - 文案改成更贴近“执行中控”的语义。
- `边界说明`
  - 合并当前“系统说明”和“契约范围”中的高价值内容。
  - 压缩成一张更轻的摘要卡，避免与核心操作争抢注意力。

## 交互规则
- 表单仍然是单页提交，不拆成多步向导，不增加分页或分步保存。
- 校验错误继续贴近对应字段展示，避免把错误集中到全局区域。
- 顶部流程导航根据当前输入情况展示轻量状态：
  - 简历为空时显示待补充
  - 岗位为空时显示待补充
  - 参数区默认显示已配置
  - 提交区根据 `GenerationStatus` 显示等待输入、上传中、提交失败等状态
- “生成前确认”卡只做摘要，不新增开关、审批、确认弹窗等行为。

## 响应式策略
- `xl` 及以上：
  - 采用稳定双栏布局。
  - 右栏保持 `sticky`，在用户滚动左侧输入区时始终可见。
- `xl` 以下：
  - 右栏取消固定定位，回到自然流布局。
  - 页面顺序改为：流程导航、输入模块、参数、提交、状态、边界说明。
- 所有断点都保留当前可读性优先的边框、阴影和暖色背景，不引入新的主题方向。

## 保留项
- 不修改 `/api/interview-packs/create` 的调用方式。
- 不修改 `INTERVIEW_PACK_SESSION_STORAGE_KEY`、成功响应结构或结果页跳转路径。
- 不改变现有校验规则、错误文案和上传中禁用按钮逻辑。
- 不重做整体视觉风格，只做层级、排序和密度上的重构。

## 非目标
- 不把工作台改造成多页向导。
- 不新增自动保存、草稿恢复或分段提交。
- 不在本次改动中重写全局设计系统或大规模替换基础组件。

## 验收标准
- 桌面端用户能沿着“顶部步骤提示 -> 左侧填写 -> 右侧提交”的稳定路径完成一次提交。
- 右侧执行栏在桌面端滚动过程中保持可见，不遮挡主输入区。
- 左侧只出现输入和确认类信息，右侧只出现执行和状态类信息。
- 现有测试依旧覆盖基础校验、提交流程和成功跳转行为。
